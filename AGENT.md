# AGENT.md

このプロジェクトは Vite + three.js + WebGL のブラウザゲーム
「職をつぐ者 〜ジパング復興編〜」（ドラゴンクエストビルダーズ・オマージュ）です。
実装の全体像・操作方法は [README.md](README.md) を参照してください。

## 最優先ルール

- ユーザーから「コード修正しないでください」と指示された場合は、コード・アセット・設定ファイルを変更しない。
- 既知の不具合が明示されている場合、同じ確認を最初からやり直さず、対象箇所の整理・設計・修正に集中する。
- ユーザーがスクリーンショットや動作結果を提示している場合、それを一次情報として扱う。

## ブラウザ確認のルール

このプロジェクトの動作確認で、自前のブラウザ自動化を組まない。

使ってよいもの:

- in-app の Preview / Browser 系ツール（`preview_start` / `preview_screenshot` / `preview_eval` など）
- ユーザーが提示したスクリーンショットと動作報告

やってはいけないこと:

- PowerShell で Edge / Chrome を直接起動して確認環境を作る
- `msedge --headless --screenshot` の一発撮り
- DevTools Protocol / CDP を自作してスクリーンショットを撮る
- `file:///dist/index.html` を開いて WebGL の確認をする
- ポート番号や remote debugging port を総当たりする
- 既存ブラウザプロセスや dev server をむやみに kill する
- 起動・撮影・キー入力を1つの PowerShell コマンドに詰め込む

## dev server の扱い

- `http://localhost:5173/` がすでに動いている場合は再利用する（`.claude/launch.json` の `dev` 構成）。
- dev server は常駐させる。撮影用コマンドの中だけで一時起動しない。
- サーバー起動が必要な場合は、通常の Vite 起動方法を使う。

## プレビューが詰まったときの進め方

このプロジェクトでは、プレビュータブが hidden 状態になると `requestAnimationFrame` が止まり、
スクリーンショットがタイムアウトする・キー入力でキャラが動かない、という事象が起きやすい。
権限エラーなどで実画面確認ができない場合も含め、代替のブラウザ自動化は作らないこと。

その場合の進め方:

1. 「こちらでは実画面確認できない（または描画ループが止まっている）」と明示する。
2. `preview_eval` で **数値的に検証する**（シーン内のメッシュ位置・寸法・回転を読み取る）。
   一時的なデバッグハンドル（例: `window.__char`）を付けた場合は、確認後に必ず削除する。
3. ユーザーのスクリーンショット・動作報告・コード読解を併用して原因を切り分ける。
4. 修正後は、変更点とユーザーに確認してほしいポイントを具体的に伝える。

## アセットの再生成

頭・胴のスプライト `assets/player_body_8dir.png` は、生成元 `assets/player_image.png`
（クロマグリーン背景）から `tools/keygreen.mjs` で作る。元画像が更新されたら再生成する:

```powershell
npm.cmd install pngjs --no-save
node tools/keygreen.mjs
```

## 作業方針

- 確認に時間を使いすぎない。Preview がすぐ使えない場合は数値検証・実装・設計・ユーザー確認に戻る。
- キャラ調整は基本的に `src/main.js` 冒頭のパラメータ（[README.md](README.md) 参照）で完結させ、
  ジオメトリ生成ロジックの変更は必要なときだけ行う。
- 作業完了時は、変更内容・確認したこと・ユーザー側で確認してほしいポイントを短く報告する。
